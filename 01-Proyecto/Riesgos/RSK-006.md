# RSK-006: Pérdida de trabajo por falla en sincronización Git

## Descripción

Si el plugin obsidian-git falla silenciosamente (no hace commit/push), un integrante podría trabajar horas sin respaldo. Si luego ocurre un fallo del disco, corrupción del vault, o conflicto de merge destructivo, se perdería trabajo no respaldado.

## Causa Raíz

- obsidian-git puede fallar sin notificación visible
- El auto-commit depende de que Obsidian esté abierto
- Errores de autenticación Git pueden bloquear push/pull silenciosamente
- Los integrantes pueden no verificar que la sincronización funcione

## Impacto Detallado

| Dimensión | Descripción del Impacto |
|-----------|------------------------|
| **Alcance** | Pérdida de horas de trabajo (peor caso: 1 día completo) |
| **Calendario** | Rehacer trabajo perdido + debugging de Git |
| **Calidad** | Posible corrupción de archivos si merge falla |
| **Recursos** | Frustración y pérdida de confianza en el sistema |

## Trigger (Indicador de Alerta)

- Commits no aparecen en GitHub después de 30 minutos
- Barra de estado de Obsidian no muestra el ícono de Git activo
- Error de autenticación al hacer push/pull manual

## Plan de Respuesta

### Acciones Preventivas (Antes)

- [ ] Verificar semanalmente que obsidian-git esté haciendo commits automáticos → @Equipo 📅 2026-03-08
- [ ] Documentar procedimiento de recuperación desde GitHub si se pierde vault local 📅 2026-03-10
- [ ] Cada integrante verifica su último commit en GitHub al terminar sesión de trabajo 📅 2026-03-05

### Acciones de Contingencia (Si ocurre)

| # | Acción | Responsable | Recursos necesarios |
|---|--------|------------|---------------------|
| 1 | Clonar repositorio desde GitHub a nueva ubicación | Afectado | `git clone` |
| 2 | Recuperar archivos desde `.git` local si el repo no está corrupto | Geovanny | `git fsck` + `git reflog` |
| 3 | Si se pierde trabajo, reconstruir desde último commit en GitHub | Equipo | Historial de GitHub |
