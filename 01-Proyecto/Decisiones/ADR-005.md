# ADR-005: Frontmatter YAML estandarizado por tipo de nota

## Contexto

Para que Dataview pueda generar dashboards, RTM y métricas automáticas, todas las notas deben tener un frontmatter YAML consistente. Sin un esquema estricto, las queries fallan silenciosamente y las estadísticas son incorrectas.

> Decisión originada en [[MIN-001]].

## Decisión

**Esquemas YAML estrictos por tipo:**

| Tipo (`type:`) | Campos obligatorios | Carpeta |
|----------------|---------------------|---------|
| `task` | id, title, status, priority, assignee, sprint, module, requirement | 05-Sprints/ |
| `requirement/functional` | id, module, wbs, status, priority, actor | 03-Requerimientos/Funcionales/ |
| `requirement/non-functional` | id, category, wbs, status, priority, metric | 03-Requerimientos/No Funcionales/ |
| `meeting` | id, date, attendees, decisions, action-items, risks | 07-Reuniones/ |
| `adr` | id, title, status, category, module, impact, deciders | 01-Proyecto/Decisiones/ |
| `risk` | id, title, status, probability, impact, severity, strategy, owner | 01-Proyecto/Riesgos/ |
| `sprint-planning` | sprint, start, end, goal, status | 05-Sprints/ |

## Consecuencias

### Positivas

- Dataview queries 100% confiables
- Estadísticas automáticas y correctas
- Trazabilidad bidireccional garantizada
- Validable programáticamente

### Negativas / Riesgos

- [[RSK-005]]: Errores si se crean notas sin template
- Mantenimiento del esquema cuando se agregan campos nuevos

## Riesgos Relacionados

- [[RSK-005]] — Complejidad del frontmatter si se crean notas manualmente
