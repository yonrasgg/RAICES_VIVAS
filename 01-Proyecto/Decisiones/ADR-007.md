# ADR-007: Migrar Gestión de Proyecto a Jira Cloud

## Estado

� Aceptada — Opción D (jira-sync + obsidian-jira-issue)

## Contexto

Raíces Vivas utiliza actualmente un vault de Obsidian (`RAICES_VIVAS`) como hub central de documentación y gestión de proyecto. El seguimiento de tareas se realiza mediante:

- Frontmatter YAML en archivos `.md` (T-001..T-031)
- Kanban plugin de Obsidian (Backlog.md)
- DataviewJS para métricas y dashboards

Sin embargo, el equipo necesita:

1. **Visibilidad en tiempo real** del progreso de sprints para todos los stakeholders
2. **Board Scrum visual** con drag-and-drop y swimlanes
3. **Reportes nativos** (burndown, velocity, cumulative flow)
4. **Integración institucional** — CENFOTEC utiliza Atlassian como ecosistema estándar

**Instancia Jira asignada:** `https://ucenfotec-team-y6xzvduw.atlassian.net/jira/software/projects/RV/boards/1`
**Proyecto:** RV (Raíces Vivas)

## Decisión

**Adoptar Jira Cloud como herramienta de gestión de proyecto operativa**, manteniendo Obsidian como hub de documentación y conocimiento.

### Arquitectura de integración: Modelo Híbrido

```
┌─────────────────────────────────┐     ┌──────────────────────────────┐
│         OBSIDIAN VAULT          │     │        JIRA CLOUD            │
│  (Documentación & Conocimiento) │     │   (Gestión Operativa)        │
│                                 │     │                              │
│  ✍️  Requerimientos (RF/RNF)    │     │  📋 Epics                    │
│  📚 Investigación               │     │  📖 User Stories             │
│  🏗️  Arquitectura               │     │  ✅ Tasks / Subtasks         │
│  📝 Minutas                     │     │  🏃 Sprints                  │
│  📊 DataviewJS Dashboards       │◄────│  📈 Burndown / Velocity      │
│  ⚠️  Riesgos / ADRs             │     │  👤 Assignments              │
│                                 │     │                              │
│  Plugin: jira-sync (escritura)  │◄──►│  REST API v3                 │
│  Plugin: jira-issue (lectura)   │────►│  Basic Auth (email + token)  │
└─────────────────────────────────┘     └──────────────────────────────┘
```

**Principio clave:** Jira es la **fuente de verdad** para estado operativo (status, assignee, sprint). Obsidian es la **fuente de verdad** para documentación (descripción, criterios, contexto).

## Alternativas Consideradas

### Opción A — Solo Plugin (read-only)
- Instalar `obsidian-jira-issue` → dashboards JQL en Obsidian
- ✅ Zero code, setup rápido
- ❌ No sincroniza frontmatter, gestión totalmente en Jira

### Opción B — Solo Script (bidireccional)
- Python `jira_sync.py` → sync frontmatter ↔ Jira REST API
- ✅ Automatización completa
- ❌ Complejidad de mantenimiento, conflictos de merge

### Opción C — Híbrido (script + plugin lectura)
- Plugin para dashboards live en Obsidian
- Script Python para carga inicial (batch) y sincronización puntual
- Field `jira_key` en frontmatter vincula nota ↔ issue
- ✅ Lo mejor de ambos mundos
- ❌ Script custom que mantener

### Opción D — Doble Plugin ⭐ ELEGIDA
- **jira-sync** (Alamion): sync bidireccional — crear, actualizar, status, worklog
- **obsidian-jira-issue** (marc0l92): dashboards JQL inline, contadores, badges
- Sin script custom permanente (solo batch migration si necesario)
- ✅ Sync completo desde Obsidian, cero código a mantener
- ✅ Ambos plugins en Community Plugins (1-click install)
- ✅ jira-sync soporta Basic Auth (Jira Cloud compatible)
- ✅ Field mapping avanzado (cualquier campo, incluso custom)

### Fase 3 — Crear Dashboards en Obsidian (🤖 AUTOMATIZABLE)

#### 3.1 Dashboard Sprint Actual

En `00-Dashboard/Home.md`, agregar bloque:

````markdown
## 🏃 Sprint Activo — Jira

```jira-search
type: TABLE
query: project = RV AND sprint in openSprints() ORDER BY priority DESC, status ASC
columns: key, summary, status, assignee, priority
```
````

#### 3.2 Dashboard por Módulo

En cada nota de módulo o en Métricas.md:

````markdown
### Módulo Educación

```jira-search
type: TABLE
query: project = RV AND component = educacion ORDER BY status ASC
columns: key, summary, type, status, assignee
```

### Conteo de issues abiertos

```jira-count
query: project = RV AND status != Done
```
````

#### 3.3 Inline Issues en Notas

En cualquier nota .md, referenciar un issue:

```markdown
La tarea `JIRA:RV-15` implementa el registro de docentes.
```

### Fase 4 — Vincular Frontmatter con Jira (🤖 + 🧑)

#### 4.1 Agregar `jira_key` al Frontmatter

Una vez creados los issues en Jira, agregar campo `jira_key` a cada nota correspondiente:

```yaml
---
type: task
id: T-021
jira_key: RV-15        # ← NUEVO: vincula con issue Jira
title: "Diseñar diagrama de contexto del sistema"
status: todo
# ...resto del frontmatter
---
```

Esto permite:
- Buscar en vault por `jira_key`
- DataviewJS puede mostrar el link a Jira
- El script de sync puede usarlo para sincronización

#### 4.2 Script de Carga Inicial (Opcional)

Se puede crear `08-Recursos/scripts/jira_sync.py` que:
1. Lee todos los archivos con `type: task` o `type: requirement/*`
2. Crea issues en Jira vía REST API (bulk create)
3. Escribe el `jira_key` resultante de vuelta en el frontmatter
4. Mapea sprints, prioridades, assignees automáticamente

> **Nota:** Este script es OPCIONAL. Si el equipo prefiere, se pueden crear los issues manualmente en Jira y luego agregar `jira_key` a mano. Con ~55 issues (4 Epics + 23 Stories + 31 Tasks), hacerlo manualmente toma ~2-3 horas.

### Fase 5 — Workflow Diario

```
📝 Documentación            📋 Gestión Operativa
    Obsidian                      Jira
       │                           │
       │  Escribir specs,          │  Crear/mover tasks,
       │  investigación,           │  asignar sprints,
       │  minutas, ADRs            │  actualizar status
       │                           │
       └─────────┬─────────────────┘
                 │
          jira_key vincula
          ambos mundos
                 │
       ┌─────────┴─────────────────┐
       │                           │
   DataviewJS                  Burndown
   en Obsidian                 en Jira
   muestra estado              muestra progreso
```

**Reglas de workflow:**

1. **Status se actualiza en Jira** (drag-and-drop en board)
2. **Documentación se actualiza en Obsidian** (descripción, criterios, evidencia)
3. **Sprint planning** se hace en Jira (asignar stories a sprint, priorizar)
4. **Sprint review** se documenta en Obsidian (minuta + retrospectiva)
5. `jira_key` es el puente entre ambos sistemas

## Consecuencias

### Positivas
- Board visual Scrum para todo el equipo y stakeholders
- Reportes nativos (burndown, velocity) sin configuración manual
- Búsqueda y filtrado avanzado con JQL
- Integración con ecosistema Atlassian (Confluence si se necesita)
- Dashboards live en Obsidian via plugin
- Obsidian mantiene su rol como base de conocimiento

### Negativas
- Duplicación parcial de datos (status en Jira + status en frontmatter)
- Dependencia de conectividad para Jira (Obsidian sigue offline-first)
- Curva de aprendizaje inicial para el equipo
- Necesidad de mantener `jira_key` sincronizado

### Mitigaciones
- `jira_key` en frontmatter como single source of link
- DataviewJS puede leer frontmatter ; JQL puede leer Jira → no hay conflicto de fuentes
- Status en Obsidian se vuelve "informativo" ; Jira es autoritativo
- Sprint planning/review keepan tanto Jira como Obsidian actualizados
