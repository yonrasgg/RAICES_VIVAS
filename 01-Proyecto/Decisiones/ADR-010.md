## Contexto

El equipo necesitaba una forma estándar de crear diagramas de arquitectura (C4), modelos entidad-relación y otros diagramas técnicos directamente dentro del vault de Obsidian. Se requerían diagramas que fueran:

- Versionables en Git (texto plano, no binarios)
- Renderizables dentro de Obsidian sin herramientas externas
- Editables por todos los integrantes sin software adicional

> Decisión originada en [[MIN-002]].

## Opciones Consideradas

| # | Opción | Pros | Contras |
|---|--------|------|---------|
| 1 | **Mermaid** | Nativo en Obsidian, texto plano, soporta ER + C4 + flowchart | Limitado en diseño visual, curva sintaxis |
| 2 | **draw.io / diagrams.net** | UI visual, exporta PNG/SVG | Archivos binarios, difícil de versionar en Git |
| 3 | **PlantUML** | Potente para UML | Requiere servidor externo o Java local |
| 4 | **Excalidraw** | Visual e informal, nativo en Obsidian | No orientado a diagramas técnicos formales |

## Decisión

**Opción seleccionada: Mermaid**

Mermaid se adopta como formato oficial para todos los diagramas técnicos del proyecto porque:
1. **Nativo en Obsidian** — se renderiza sin plugins adicionales
2. **Texto plano** — se versiona perfectamente en Git y se puede hacer diff
3. **Soporta los tipos necesarios:** `erDiagram`, `C4Context`, `flowchart`, `sequenceDiagram`
4. **Consistencia** — un solo formato para todo el equipo

Excalidraw se reserva para wireframes y bocetos informales ([[ADR-012]]).

## Consecuencias

### Positivas
- Todos los diagramas son texto plano dentro del vault
- Renderizado instantáneo en vista Preview de Obsidian
- Historial de cambios completo en Git para cada diagrama

### Negativas / Riesgos
- La complejidad visual de diagramas grandes puede ser difícil de manejar
- Algunos tipos de diagramas (como C4 Container) tienen soporte limitado en Mermaid

### Compromisos
- Se prioriza versionabilidad sobre estética visual
