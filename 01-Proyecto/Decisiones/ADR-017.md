# ADR-017: Migración de Dataview a SQLSeal + SQLSeal Charts

## Estado

**Accepted** — 2026-03-27

## Contexto

El vault RAICES_VIVAS utiliza **286 bloques Dataview** (173 DV + 45 DVJS + 68 inline) distribuidos en 51 archivos y 17 templates. Dataview presenta limitaciones:

1. **Lenguaje propietario**: DQL no es SQL estándar — curva de aprendizaje innecesaria
2. **No soporta CSV nativamente**: datos tabulares como finanzas y encuestas requieren soluciones ad-hoc
3. **DVJS frágil**: bloques JavaScript embebido son difíciles de mantener y auditar
4. **Sin gráficos nativos**: requiere plugins adicionales (Charts) con sintaxis independiente
5. **Rendimiento**: escaneo completo del vault en cada query, sin caché SQL

**SQLSeal** (v0.39.1) ofrece:

- SQL estándar sobre tabla virtual `files` (frontmatter como columnas dinámicas)
- Carga directa de CSV con `TABLE x = file('path.csv')`
- JOINs entre frontmatter y CSVs
- 5 formatos de salida: GRID, HTML, MARKDOWN, LIST, TEMPLATE (Lodash)
- Plugin companion **SQLSeal Charts** (v0.4.2) con ECharts 5.6.0 integrado

## Decisión

Migrar **todos los bloques Dataview** a SQLSeal SQL, en 5 fases:

| Fase | Descripción | Bloques |
|------|-------------|---------|
| 0 | Infraestructura: CSVs, script de extracción, tests | — |
| 1 | Bloques triviales (RF, Stories, Sprint Planning) | ~50 |
| 2 | Dashboards complejos (Métricas, Finanzas, Home) | ~45 |
| 3 | Templates y Daily/Weekly notes | ~100 |
| 4 | Inline DV → Templater + eliminación final de Dataview | ~68 |

### Compatibilidades preservadas

- **Meta Bind**: INPUT[]/VIEW[] siguen escribiendo/leyendo YAML frontmatter
- **JIRA Sync**: 14 field mappings sin cambios (frontmatter → Jira REST API)
- **Templater**: Absorbe funciones de auto-ID que hoy hacen inline DV expressions

### Datos tabulares como CSV

| CSV | Contenido | Regenerable |
|-----|-----------|-------------|
| `finanzas-config.csv` | Tarifas y roles | Manual |
| `sprint-plan.csv` | Horas planeadas vs reales | Manual |
| `riesgos.csv` | Frontmatter de RSK-*.md | Sí (script) |
| `decisiones.csv` | Frontmatter de ADR-*.md | Sí (script) |

## Alternativas Consideradas

### 1. Mantener Dataview (status quo)

- ✅ Sin esfuerzo de migración
- ❌ Deuda técnica creciente con DVJS
- ❌ Sin SQL estándar, sin JOINs con CSV

### 2. DB Folder

- ✅ Interfaz visual de tabla
- ❌ No soporta SQL
- ❌ No permite gráficos

### 3. SQLSeal (elegido)

- ✅ SQL estándar
- ✅ CSV nativo + JOINs
- ✅ Charts integrados (ECharts)
- ✅ TEMPLATE output para formateo rico
- ❌ Esfuerzo de migración (~286 bloques)
- ❌ Sin inline queries (reemplazado por Templater)

## Consecuencias

### Positivas

- SQL estándar transferible a otros contextos
- Datos tabulares (encuestas, finanzas) consultables directamente
- Gráficos dinámicos sin plugin adicional separado
- Reducción de plugins activos (−Dataview, −Charts separado)

### Negativas

- Esfuerzo estimado de migración: 5 fases
- Período transitorio con ambos plugins activos
- Inline expressions requieren conversión a Templater

### Riesgos

- **RSK-005** (complejidad frontmatter): Mitigado — YAML permanece igual, solo cambia el motor de consulta
