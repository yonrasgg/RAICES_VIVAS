# ADR-008: Selección de Stack Tecnológico para Raíces Vivas

## Contexto

Raíces Vivas requiere una plataforma tecnológica que funcione en **territorios indígenas de Costa Rica** con las siguientes restricciones operativas:

- **Conectividad intermitente o nula** en zonas como Talamanca, Alto Chirripó, Buenos Aires
- **Dispositivos de gama baja:** Tablets Android 8+ (Samsung Galaxy Tab A8, Lenovo Tab M10)
- **Multilingüe:** Interfaz en Español, Bribri, Cabécar y Ngäbe como mínimo
- **Usuarios no técnicos:** Docentes comunitarios, auxiliares de salud, líderes culturales
- **Seguridad de datos sensibles:** Historiales médicos (Ley 8968 Protección de Datos), saberes ceremoniales restringidos

El stack debe soportar los 3 módulos (EDU, SAB, SAL) + funcionalidad transversal (sincronización offline, multilingüe, gestión de roles).

## Opciones Consideradas

| # | Opción | Descripción | Pros | Contras |
|---|--------|-------------|------|---------|
| 1 | **PWA (React + Workbox + PouchDB)** | Progressive Web App con Service Workers para offline y PouchDB para sincronización | Offline-first nativo, un solo codebase, despliegue web estándar, comunidad amplia, gratuito | Acceso limitado a hardware nativo, rendimiento inferior en dispositivos muy viejos |
| 2 | **Hybrid (Flutter + SQLite + Hive)** | Aplicación híbrida compilada para Android/iOS con almacenamiento local | Rendimiento nativo, acceso completo a hardware, UI consistente | Mayor curva de aprendizaje, compilación necesaria, más complejo de desplegar |
| 3 | **Low-Code (AppSheet / Power Apps)** | Plataforma sin código con conectores a Google Sheets/DB | Velocidad de desarrollo, sin necesidad de programadores, rápido prototipado | Sin capacidad offline real, dependencia de proveedor, limitado en personalización, costos de licencia |

## Análisis FODA por Opción

### Opción 1: PWA (React + Workbox + PouchDB)

| | Factores Internos | Factores Externos |
|---|---|---|
| **Positivos** | **Fortalezas:** Service Workers para operación offline, PouchDB/CouchDB con conflict-resolution nativo, i18next para multilingüe, React ecosystem maduro, gratuito | **Oportunidades:** PWA cada vez más soportada en Android, Google incentiva adoption, comunidad activa de desarrolladores |
| **Negativos** | **Debilidades:** No accede a todas las APIs nativas del dispositivo, rendimiento menos predecible en gama baja, requiere HTTPS para Service Worker | **Amenazas:** Fragmentación de navegadores en Android viejo, APIs de Service Worker pueden cambiar |

### Opción 2: Hybrid (Flutter + SQLite + Hive)

| | Factores Internos | Factores Externos |
|---|---|---|
| **Positivos** | **Fortalezas:** Compilación nativa = rendimiento óptimo, acceso completo a cámara/GPS/storage, Hot Reload para desarrollo rápido | **Oportunidades:** Flutter adoption creciente, Google backing, buena documentación |
| **Negativos** | **Debilidades:** Curva de aprendizaje Dart, equipo no tiene experiencia Flutter, sync offline requiere implementación manual | **Amenazas:** Dependencia de Google, tamaño de APK mayor (~20MB+) |

### Opción 3: Low-Code (AppSheet)

| | Factores Internos | Factores Externos |
|---|---|---|
| **Positivos** | **Fortalezas:** Desarrollo sin código, prototipado inmediato, integración con Google Workspace | **Oportunidades:** Ideal para validación rápida de concepto |
| **Negativos** | **Debilidades:** Sin capacidad offline real para datos complejos, personalización muy limitada, no soporta i18n nativo | **Amenazas:** Costos de licencia ($5-$10/usuario/mes), vendor lock-in total, no funciona sin internet estable |

## Decisión

> **Estado: PROPUESTA — Pendiente de aprobación formal del equipo.**

**Opción recomendada:** **Opción 1 — PWA (React + Workbox + PouchDB/CouchDB)**

### Justificación

1. **Offline-first nativo:** PouchDB con CouchDB backend provee sincronización bidireccional con conflict resolution automático, crítico para territorios sin conectividad
2. **Multilingüe:** i18next es el estándar de facto para internacionalización en React, con soporte para archivos JSON por lengua
3. **Costo cero:** Todo el stack es open source y gratuito
4. **Equipo compatible:** El equipo tiene experiencia previa en JavaScript/React
5. **Despliegue simple:** Una URL + Service Worker, sin necesidad de Play Store

### Stack Técnico Propuesto

| Capa | Tecnología | Justificación |
|------|-----------|---------------|
| Frontend | React 18 + TypeScript | Ecosystem maduro, tipado estático |
| UI Framework | Tailwind CSS + Headless UI | Ligero, personalizable, mobile-first |
| Offline Storage | PouchDB | Sync nativo con CouchDB, conflict resolution |
| Backend DB | CouchDB | Replicación master-master, ideal para sync offline |
| Service Workers | Workbox (Google) | Estrategias de cache probadas |
| Internacionalización | i18next + react-i18next | Soporte JSON por lengua |
| Autenticación | JWT + bcrypt | Stateless, funciona offline |
| API | Node.js + Express | JavaScript full-stack |
| Multimedia | WebP (imágenes) + Opus (audio) | Compresión eficiente para bajo bandwidth |

## Consecuencias

### Positivas

- Un solo codebase para web y móvil (PWA installable)
- Funcionamiento garantizado sin internet después de primera carga
- Costos de desarrollo y mantenimiento mínimos
- Facilidad de actualización (deploy web, no app store)
- Compatibilidad con el modelo de sync para servidores locales (Raspberry Pi en centros comunales)

### Negativas / Riesgos

- Performance en Android 8 con mucho contenido multimedia puede degradarse
- No accede a todas las APIs nativas (e.g., Bluetooth, NFC)
- Requiere HTTPS (necesita certificado SSL incluso para servidor local)

### Compromisos (Trade-offs)

- Se sacrifica rendimiento nativo (Flutter) por simplicidad y costo
- Se sacrifica velocidad de desarrollo (Low-Code) por capacidades offline reales
- Se acepta limitación de APIs nativas a cambio de un solo codebase web
