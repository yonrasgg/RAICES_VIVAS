# ADR-003: GitHub Projects para gestión ágil en fases posteriores

## Contexto

Para la fase de implementación se necesitará un tablero Kanban más robusto con automatizaciones CI/CD. GitHub Projects puede complementar Obsidian sin reemplazarlo, ofreciendo vistas de proyecto para stakeholders que no usen Obsidian.

> Decisión originada en [[MIN-001]].

## Opciones Consideradas

| # | Opción | Pros | Contras |
|---|--------|------|---------|
| 1 | **GitHub Projects** | Integrado con repos, vistas múltiples, automatización con Actions | Duplicación potencial con Kanban de Obsidian |
| 2 | **Solo Obsidian Kanban** | Un solo sistema | Limitado en automatización CI/CD |
| 3 | **Trello** | Familiar, visual | Otro sistema más que mantener, sin integración Git |

## Decisión

**Opción seleccionada: GitHub Projects (implementación futura)**

- Se activará cuando inicie la fase de desarrollo/implementación
- Obsidian permanece como fuente de verdad para documentación
- GitHub Projects será la vista de ejecución para sprints de código

## Consecuencias

### Positivas

- Vistas de proyecto para stakeholders sin acceso a Obsidian
- Automatización con GitHub Actions
- Integración nativa con pull requests y issues

### Negativas / Riesgos

- [[RSK-004]]: Riesgo de inconsistencia entre Obsidian y GitHub Projects
- Esfuerzo adicional de configuración y sincronización manual
