**Fecha:** 2026-02-25

---

## Agenda

1. Definición del enfoque de gestión del proyecto
2. Selección de herramientas de trabajo colaborativo
3. Definición de requerimientos funcionales y no funcionales
4. Estructura del vault y convenciones

## Discusión

### 1. Sistema de Gestión del Proyecto

Se evaluaron las opciones para la gestión del proyecto. Se decidió usar **Obsidian** como centro de control único, manteniendo la filosofía de:

- **Orden**: Estructura de carpetas numeradas (00-Dashboard → 09-QA → 99-Templates)
- **Estructura**: Frontmatter estandarizado por tipo de nota (requerimiento, tarea, riesgo, minuta)
- **Formato**: Markdown puro, consultas Dataview dinámicas, templates con Templater

> 📌 Obsidian será siempre el centro de control del proyecto. Todas las métricas, trazabilidad y seguimiento se gestionan desde el vault.

### 2. Control de Versiones y Colaboración

Se decidió usar **Git + GitHub** para la sincronización entre los 3 integrantes:

- **Repositorio privado:** `github.com/yonrasgg/RAICES_VIVAS`
- **Plugin Obsidian Git:** Auto commit/push/pull cada 10 minutos
- **Credenciales:** Cada integrante usa un Personal Access Token (PAT)
- **Se descartó Google Drive** por problemas de conflictos silenciosos y falta de historial granular

### 3. Gestión Ágil — Roadmap

Se acordó que la gestión ágil se implementará de forma progresiva:

| Fase | Herramienta | Alcance |
|------|-------------|---------|
| **Actual** | Obsidian (Kanban + Tasks + Dataview) | Sprints, tareas, trazabilidad completa |
| **Futura** | GitHub Projects + Milestones | Tableros de proyecto, automatización de issues, integración CI/CD |

> ⚠️ La migración a GitHub Projects se hará manteniendo Obsidian como fuente de verdad para documentación y trazabilidad. GitHub Projects complementará con vistas de proyecto y automatización.

### 4. Requerimientos del Sistema

Se definieron los requerimientos del sistema "Raíces Vivas — Sistema Integral de Apoyo a Comunidades Indígenas":

**Requerimientos Funcionales (16 RF):**

| Módulo | IDs | Cantidad |
|--------|-----|----------|
| Educación (EDU) | [[RF-EDU-01]] a [[RF-EDU-06]] | 6 |
| Saberes Ancestrales (SAB) | [[RF-SAB-01]] a [[RF-SAB-05]] | 5 |
| Salud (SAL) | [[RF-SAL-01]] a [[RF-SAL-05]] | 5 |

**Requerimientos No Funcionales (7 RNF):**
- [[RF-TRANS-01]] a [[RF-TRANS-03]] (rendimiento, seguridad, accesibilidad, escalabilidad, etc.)

Cada requerimiento fue documentado como nota individual con frontmatter estructurado, criterios de aceptación y trazabilidad.

### 5. Estructura del Vault

Se definió la estructura de carpetas del vault:

```
RAICES_VIVAS/
├── 00-Dashboard/        ← Métricas y KPIs dinámicos
├── 01-Proyecto/         ← Charter, Workflow, Onboarding
├── 02-Investigación/    ← Contexto, fuentes, hallazgos
├── 03-Requerimientos/   ← RF + RNF + RTM (Matriz de Trazabilidad)
├── 04-Arquitectura/     ← Diagramas, ADRs
├── 05-Sprints/          ← Tareas por sprint
├── 06-Entregables/      ← Documentos de entrega
├── 07-Reuniones/        ← Minutas (esta nota)
├── 08-Recursos/         ← Imágenes, glosario
├── 09-QA/               ← Control de calidad
├── 99-Templates/        ← Plantillas estandarizadas
└── Daily Notes/         ← Notas diarias
```
