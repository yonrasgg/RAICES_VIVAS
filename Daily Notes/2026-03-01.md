## 📊 Métricas del Día

```sqlseal
SELECT
  SUM(CASE WHEN status = 'done' AND completed = @date THEN 1 ELSE 0 END) as "✅ Hoy",
  SUM(CASE WHEN status IN ('in-progress', 'review') THEN 1 ELSE 0 END) as "🔄 En curso",
  SUM(CASE WHEN status = 'blocked' THEN 1 ELSE 0 END) as "🚫 Bloqueadas",
  SUM(CASE WHEN status = 'done' THEN 1 ELSE 0 END) || '/' || COUNT(*) || ' (' || ROUND(100.0 * SUM(CASE WHEN status = 'done' THEN 1 ELSE 0 END) / MAX(1, COUNT(*))) || '%)' as "📈 Progreso"
FROM files
WHERE (type = 'task' OR type = 'subtask') AND path LIKE '05-Sprints%'
```

### 🔧 Error del plugin Linter — Diagnóstico y corrección

Se detectó que el **plugin Linter** (específicamente la regla `yaml-key-sort`) estaba desordenando el frontmatter YAML de múltiples notas al guardar. El Linter aplicaba una única lista de prioridad de keys a **todos** los tipos de nota, pero el vault tiene tipos con órdenes canónicos incompatibles entre sí (por ejemplo, las tareas ponen `banner_src` primero mientras que los documentos ponen `type` primero).

**Consecuencias:**
- ~95 archivos con frontmatter desordenado
- 9 archivos con delimitadores `---` duplicados
- Propiedades inválidas en varias notas (visible como "Invalid properties" en Obsidian)

**Acciones tomadas:**
1. Se desactivó `yaml-key-sort` en la configuración del Linter (no es compatible con múltiples órdenes por tipo)
2. Se ejecutó un script de Python (`fix_frontmatter.py`) que normaliza el frontmatter según el tipo de cada nota
3. Se repararon los 9 archivos con delimitadores rotos
4. Se agregó `banner_src_x: 0.47714` a todos los archivos que lo tenían faltante
5. Se revirtieron cambios dañados y se actualizó la rama `main` con todas las correcciones

El script `fix_frontmatter.py` queda disponible en `08-Recursos/scripts/` para futuras normalizaciones.
